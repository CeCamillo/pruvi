# Pruvi — Native App Architecture

> Context document for Claude Code. Read this before working on anything inside `apps/native/`.

---

## Stack

| Layer | Library | Version |
|---|---|---|
| Framework | Expo | SDK 54 |
| Navigation | Expo Router | v4 (file-based) |
| JS Engine | Hermes | bundled with Expo |
| Server state | TanStack Query | v5 |
| Client state | Zustand | v5 |
| Animations | Reanimated | v3 |
| Lists | FlashList | v1 |
| Forms | react-hook-form + zod | via `@pruvi/shared` |
| Auth client | Better Auth client | self-hosted |

## Navigation Structure

Expo Router uses file-based routing. The full nav tree:

```
apps/native/app/
├── _layout.tsx                  ← Root layout: auth guard, QueryClient, theme provider
├── (auth)/
│   ├── _layout.tsx              ← Unauthenticated stack navigator
│   ├── login.tsx
│   └── register.tsx
└── (app)/
    ├── _layout.tsx              ← Drawer layout (authenticated only)
    ├── (tabs)/
    │   ├── _layout.tsx          ← Bottom tab bar
    │   ├── index.tsx            ← Home / Daily session entry point
    │   ├── progress.tsx         ← Stats & subject history
    │   └── profile.tsx          ← User profile + settings
    ├── session/
    │   ├── [id].tsx             ← Active session screen (Q&A loop)
    │   └── result.tsx           ← Session result + XP breakdown
    └── subject/
        └── [slug].tsx           ← Subject detail + review history
```

### Auth Guard

The root `_layout.tsx` checks the Better Auth session on mount. If no valid session exists, it redirects to `/(auth)/login`. No screen inside `(app)/` is ever accessible unauthenticated — enforce this at the root layout, not at the screen level.

## State Management

**Golden rule: TanStack Query owns server state. Zustand owns ephemeral client state.**

### TanStack Query (server state)

All data that lives on the backend goes through Query hooks:

```typescript
// hooks/useSessionQuery.ts
export function useTodaySession() {
  return useQuery({
    queryKey: ['session', 'today'],
    queryFn: sessionService.getToday,
    staleTime: 1000 * 60 * 5, // 5 min — pre-generated by BullMQ, no need to refetch constantly
  })
}
```

Key query hooks:
- `useTodaySession()` — fetches today's pre-generated session (`GET /sessions/today`)
- `useProgress()` — subject-level accuracy stats and review history
- `useProfile()` — user XP, streak count, character state

### Zustand (client state)

Ephemeral state that doesn't belong on the server — reset when the session ends:

```typescript
// stores/sessionStore.ts
interface SessionStore {
  currentQuestionIndex: number
  selectedOption: string | null
  answerState: 'idle' | 'correct' | 'wrong'
  livesRemaining: number
  actions: {
    selectOption: (optionId: string) => void
    submitAnswer: () => void
    nextQuestion: () => void
    reset: () => void
  }
}
```

```typescript
// stores/gamificationStore.ts
interface GamificationStore {
  pendingXP: number             // XP earned, not yet animated
  streakAnimationTrigger: boolean
  actions: {
    addXP: (amount: number) => void
    triggerStreakAnimation: () => void
    flush: () => void
  }
}
```

## Component Architecture

```
apps/native/components/
├── session/
│   ├── QuestionCard.tsx         ← Question text + 4 option buttons
│   ├── OptionButton.tsx         ← Animated: idle → selected → correct/wrong
│   ├── LivesBar.tsx             ← Heart icons, shake animation on loss
│   └── ProgressBar.tsx         ← Questions answered / total
├── gamification/
│   ├── XPCounter.tsx            ← Animated spring number counter
│   ├── StreakBadge.tsx          ← Flame icon + streak count
│   └── CharacterAvatar.tsx     ← Character with expression states
├── common/
│   ├── Screen.tsx               ← SafeAreaView + scroll wrapper (use this, not raw SafeAreaView)
│   ├── Button.tsx               ← Primary/secondary/ghost variants
│   └── Skeleton.tsx             ← Loading placeholder
└── subject/
    └── SubjectCard.tsx          ← FlashList item for subject list
```

### Component Rules

- **Always `memo()`** components that receive stable props, especially `OptionButton` and `SubjectCard`
- **Always `useCallback`** on handlers passed as props
- **Never put animation logic in component state** — use Reanimated shared values

## Animation Patterns (Reanimated v3)

**Rule: all animations run on the UI thread via worklets. Zero `setState` inside animation callbacks.**

### OptionButton (most complex animation)

```
Idle
  → tap: spring scale to 0.97 (press feedback)
  → selected: background color transition, checkmark fade in
    → correct answer revealed: spring to green, bounce scale
    → wrong answer: horizontal shake worklet (withSequence)
```

### XPCounter (session result screen)

```typescript
// Animated counter from 0 → earned XP
const xpValue = useSharedValue(0)

useEffect(() => {
  xpValue.value = withTiming(earnedXP, {
    duration: 1200,
    easing: Easing.out(Easing.cubic),
  })
}, [earnedXP])
```

### LivesBar (wrong answer)

```typescript
// Horizontal shake
translateX.value = withSequence(
  withTiming(-8, { duration: 60 }),
  withTiming(8, { duration: 60 }),
  withTiming(-8, { duration: 60 }),
  withTiming(0, { duration: 60 }),
)
```

## API Service Layer

Never call `fetch` directly from components or hooks. All API calls go through the service layer:

```
apps/native/services/
├── session.service.ts     ← getToday, startSession, answerQuestion, completeSession
├── progress.service.ts    ← getSubjectStats, getReviewHistory
└── auth.service.ts        ← login, register, logout (wraps Better Auth client)
```

All services return typed responses. The types are inferred from shared Zod schemas:

```typescript
import { sessionResponseSchema } from '@pruvi/shared/schemas'

type SessionResponse = z.infer<typeof sessionResponseSchema>

async function getToday(): Promise<SessionResponse> {
  const res = await authClient.fetch('/api/sessions/today')
  return sessionResponseSchema.parse(await res.json())
}
```

## Shared Zod Schemas

The mobile app imports validation schemas directly from `@pruvi/shared`. This is the single source of truth — backend validates with the same schema the frontend uses.

```typescript
import { answerQuestionSchema, sessionResponseSchema } from '@pruvi/shared/schemas'

// In forms: react-hook-form + zod resolver
const form = useForm({
  resolver: zodResolver(answerQuestionSchema),
})
```

**Never duplicate schema definitions in the native app.** If a schema doesn't exist in `@pruvi/shared`, add it there.

## Auth Flow

Better Auth client handles session persistence via Expo SecureStore.

```
App launch
  → Root _layout.tsx checks session (Better Auth useSession hook)
    → No session   → redirect to /(auth)/login
    → Valid session → render (app)/ tree
      → Auth client attaches token to every request automatically
      → 401 response → clear session → redirect to login
```

The auth client is initialized once in `apps/native/lib/auth-client.ts` and imported wherever needed. Never instantiate it more than once.

## Screen Breakdown

### Home (`(tabs)/index.tsx`)
- Fetches today's pre-generated session via `useTodaySession()`
- Shows streak badge, lives (hearts), subject breakdown for the day
- "Start Session" → pushes to `session/[id]`
- Empty state: "Você está em dia!" + time until next review

### Active Session (`session/[id].tsx`)
- Drives the Q&A loop via `useSessionStore`
- Renders `QuestionCard` → user picks option → `OptionButton` animates → `POST /questions/:id/answer`
- On wrong answer: deduct life from store, trigger `LivesBar` shake
- On last question: navigate to `session/result` with session stats in route params

### Session Result (`session/result.tsx`)
- Animated XP gain via `XPCounter`
- Shows: accuracy %, questions answered, streak state (maintained/broken/extended)
- "Continuar" → pop to home, call `queryClient.invalidateQueries(['session', 'today'])`

### Progress (`(tabs)/progress.tsx`)
- `FlashList` of `SubjectCard` components with accuracy bars
- Tap subject → push to `subject/[slug]`

### Profile (`(tabs)/profile.tsx`)
- `CharacterAvatar`, XP progress bar to next level, total streak count
- Settings: push notifications toggle, paper/pen mode toggle

## Performance Rules

- **FlashList everywhere** — never use `FlatList` or `ScrollView` for lists of items
- **`staleTime: 5min` on session queries** — sessions are pre-generated by BullMQ, polling is wasteful
- **`memo()` all list items** — `SubjectCard`, `OptionButton`, anything rendered inside FlashList
- **No `require()` in render paths** — Hermes compiles to bytecode at build time, dynamic requires break this
- **SVGs via `react-native-svg`** — never PNG for icons
- **Images via Expo's asset system** — `require('./assets/...')` at module level, not inside render

## New Screen Checklist

When adding a new screen to the app, create these files in order:

1. `apps/native/app/{route}.tsx` — Expo Router screen file
2. `apps/native/hooks/use{Feature}.ts` — data fetching + business logic hook
3. `apps/native/components/{Component}.tsx` — UI component(s)
4. `apps/native/services/{feature}.service.ts` — API call layer (if new endpoint)

If the new screen requires a new API endpoint, define the Zod schema in `@pruvi/shared` first, then implement the endpoint in `apps/server`, then consume it in the native app.

---

## Key Constraints

- **Never access the database from the native app** — only via the Fastify API
- **Never duplicate Zod schemas** — always import from `@pruvi/shared/schemas`
- **Never use `any`** — TypeScript strict mode is enforced
- **Never put business logic in screens** — screens render, hooks handle logic
- **Zustand stores reset on session end** — do not persist session state to SecureStore
