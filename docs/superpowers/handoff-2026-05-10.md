## Project conventions established

### Code organization
- Feature modules at `apps/server/src/features/<name>/` with files `<name>.{repository,service,route}.ts`, `index.ts`, plus `.service.test.ts` (unit) and `.repository.integration.test.ts` (PGlite)
- Each repository is a class with explicit `select({...})` projections (no `.select()` without columns — the `unseen` branch in `questions.repository.ts` was the last exception, now fixed)
- Services wrap returns in `neverthrow` `Result<T, AppError>` with explicit error classes from `apps/server/src/utils/errors.ts`
- Routes use `fastify-type-provider-zod`, call `unwrapResult(result)` which throws `AppError` (caught by `errorHandlerPlugin`)
- Shared Zod schemas live in `@pruvi/shared/<name>.ts` (one file per feature)
- Response payloads consistently wrapped as `{ success: true, data: ... }` via `successResponse(...)`

### Caching
- Redis-backed via `fastify.cache.get/set/del`
- Cache keys are scoped: `subjects:list` (shared), `prefs:{userId}`, `progress:{userId}`, `calendar:{userId}:{from}:{to}`, `lives:{userId}`, `xp:{userId}`, `session-today:{userId}`, `streaks:{userId}`
- Invalidate on writes that affect cached data (e.g. answer endpoint invalidates lives/xp/progress)

### Schema-vs-migration discipline
- TS schemas are authoritative. `drizzle-kit generate` produces purely additive migrations
- `test-client.ts` PGlite DDL is hand-maintained — **must be updated in lockstep** with every schema change for the column being touched
- `db-helpers.ts` iterates all migrations in sorted order (not just the latest — Phase 1A fix)

### Workflow norms
- Stacked PRs branched off each preceding phase
- Brainstorm spec → write plan → subagent-driven execution → spec review + code-quality review per task
- All today's work used the `superpowers:*` skill chain
- Commit messages: conventional commits, brief, no "Claude Code" attribution
- Each PR included a design spec + implementation plan committed alongside the code (`docs/superpowers/specs/`, `docs/superpowers/plans/`)

### Known intentional deviations
- `livesResetAt` uses bare `timestamp` (not `timestamptz`) for consistency with other `user` timestamps. Server-local timezone semantics throughout (streak, progress, calendar).
- SM-2 quality currently binary-mapped (correct=4, wrong=1). Full 0-5 grading deferred.
- Lives refill: all-at-once 24h reset. Spec wants 1-per-4h — deferred to Phase 2.
- `@pruvi/auth/src/index.ts` has a `calculateSM2` 2-arg wrapper over `calculateSm2` 1-arg API — preserves call sites without forcing a service refactor.
- Apple OAuth provider is conditionally registered (only if all 3 Apple env vars set) — server boots clean without Apple Developer credentials.

## Running things locally

```bash
# Setup
pnpm install

# DB (Postgres on Docker, port 5432, db=pruvi)
docker compose -f packages/db/docker-compose.yml up -d  # via pnpm db:start

# Verify schema
pnpm verify:migration

# Tests
pnpm --filter server test                  # 70 unit tests
pnpm --filter server test:integration      # 20 integration tests
pnpm run check-types                       # turbo check-types across workspace

# Dev server
pnpm dev:server   # HTTP + Fastify
pnpm dev:worker   # BullMQ worker (separate process)

# Migration smoke
pnpm verify:migration
```

`.env` for `apps/server/.env` requires: `DATABASE_URL`, `BETTER_AUTH_SECRET` (32+ chars), `BETTER_AUTH_URL`, `CORS_ORIGIN`, `REDIS_URL`, `PORT`, `RESEND_API_KEY`, `RESEND_FROM_EMAIL`, `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`. Optional: `APPLE_CLIENT_ID`, `APPLE_CLIENT_SECRET`, `APPLE_BUNDLE_ID`.

Dev placeholder values for Resend + Google are committed in `apps/server/.env` (`replace_with_real_*`). Real credentials needed before any signup/verify flow actually works end-to-end.
